# Testing Guide

This document covers testing patterns, conventions, and how to run the test suites for the Terraform Registry frontend.

## Running Tests

All commands run from the `frontend/` directory.

```bash
# Run all unit tests once
npm test

# Run in watch mode (re-runs on file changes)
npm run test:watch

# Run with V8 coverage report
npm run test:coverage

# Run a single test file
npx vitest run src/services/__tests__/api.test.ts

# Run tests matching a pattern
npx vitest run --reporter=verbose -t "401 handling"
```

### Coverage

Coverage is generated by `@vitest/coverage-v8` and configured in `vitest.config.ts`:

| Threshold  | Minimum |
| ---------- | ------- |
| Statements | 80%     |
| Branches   | 70%     |
| Functions  | 70%     |
| Lines      | 80%     |

These thresholds are enforced in CI (`npm run test:coverage` fails the build if any threshold is not met). They will be ratcheted upward as test coverage grows.

Reports are written to `frontend/coverage/` in `text`, `text-summary`, `lcov`, and `json-summary` formats. The `lcov` report can be opened in a browser or fed into coverage visualization tools.

## Unit Test Patterns

### Mocking the API client with `vi.mock`

The API client (`services/api.ts`) is a class-based singleton. Tests mock it at the module level using `vi.mock` and `vi.hoisted`:

```ts
// vi.hoisted runs BEFORE imports, so mock variables are available in vi.mock
const mockApi = vi.hoisted(() => ({
  getModule: vi.fn(),
  getModuleVersions: vi.fn(),
  searchModules: vi.fn(),
}));

vi.mock('../../services/api', () => ({
  default: mockApi,
}));
```

This pattern is used consistently across:
- `services/__tests__/api.test.ts` -- mocks `axios` to test interceptors
- `hooks/__tests__/useModuleDetail.test.ts` -- mocks api methods
- `contexts/__tests__/AuthContext.test.tsx` -- mocks `api.getCurrentUserWithRole`, `api.login`, etc.

### Mocking Axios for interceptor tests

The API test file uses a more advanced pattern that captures the interceptor callbacks registered during `ApiClient` construction. This allows testing auth header injection and 401 handling directly:

```ts
vi.mock('axios', () => {
  const mockInstance = {
    get: vi.fn(),
    post: vi.fn(),
    interceptors: {
      request: { use: vi.fn((fulfilled) => { capturedReqFulfilled = fulfilled; }) },
      response: { use: vi.fn((fulfilled, rejected) => { /* capture handlers */ }) },
    },
  };
  return { default: { create: vi.fn(() => mockInstance) } };
});
```

Each test calls `getApiClient()` which resets modules (`vi.resetModules()`) and re-imports `api.ts` to get a fresh instance with fresh interceptors.

### Testing hooks with `renderHook`

Custom hooks that call React Query or use React context need a wrapper component providing the necessary providers:

```ts
function createWrapper() {
  const queryClient = new QueryClient({
    defaultOptions: { queries: { retry: false } },
  });
  return ({ children }: { children: React.ReactNode }) =>
    React.createElement(QueryClientProvider, { client: queryClient }, children);
}

// Usage
const { result } = renderHook(() => useModuleDetail(), {
  wrapper: createWrapper(),
});
```

Wait for async state with `waitFor`:

```ts
await waitFor(() => {
  expect(result.current.loading).toBe(false);
});
```

Trigger state changes with `act`:

```ts
act(() => {
  result.current.setSelectedVersion({ version: '1.0.0' } as any);
});
```

### Testing context providers

Context tests render a consumer component inside the provider and capture context values via a callback:

```ts
function AuthConsumer({ onRender }: { onRender: (auth: ReturnType<typeof useAuth>) => void }) {
  const auth = useAuth();
  onRender(auth);
  return <div>{String(auth.isAuthenticated)}</div>;
}

// In test:
let authState: ReturnType<typeof useAuth> | null = null;
render(
  <AuthProvider>
    <AuthConsumer onRender={(auth) => { authState = auth; }} />
  </AuthProvider>
);
expect(authState!.isAuthenticated).toBe(false);
```

### Testing components with routing

Components that depend on React Router (like `ProtectedRoute`) are wrapped in `MemoryRouter`:

```ts
function renderWithRouter(ui: React.ReactElement, { route = '/' } = {}) {
  return render(
    <MemoryRouter initialEntries={[route]}>
      <Routes>
        <Route path="/" element={ui} />
        <Route path="/login" element={<div>Login Page</div>} />
      </Routes>
    </MemoryRouter>
  );
}
```

### Mocking context hooks

When a component uses `useAuth()` internally, mock the context module rather than wrapping in a real provider:

```ts
const mockUseAuth = vi.fn();
vi.mock('../../contexts/AuthContext', () => ({
  useAuth: () => mockUseAuth(),
}));

// In test:
mockUseAuth.mockReturnValue({
  isAuthenticated: true,
  isLoading: false,
  allowedScopes: ['admin'],
});
```

## E2E Tests

### Stack

E2E tests use [Playwright](https://playwright.dev/) and run against the full application stack (frontend + backend + PostgreSQL) via Docker Compose.

Configuration: `e2e/playwright.config.ts`

| Setting  | Value                                                         |
| -------- | ------------------------------------------------------------- |
| Base URL | `https://localhost:3000` (overridable via `BASE_URL` env var) |
| Browsers | Chromium (always), Firefox (CI only)                          |
| Timeout  | 60 seconds per test                                           |
| Retries  | 0 (failures are visible immediately)                          |
| Traces   | Always recorded                                               |
| Video    | Always recorded                                               |


### Running E2E Tests Locally

1. Start the test stack:
   ```bash
   cd deployments
   docker compose -f docker-compose.test.yml up -d --build
   ```

2. Wait for the frontend to be ready on `https://localhost:3000`.

3. Install Playwright and run:
   ```bash
   cd e2e
   npm ci
   npx playwright install chromium
   npx playwright test --workers=1
   ```

4. View the HTML report:
   ```bash
   npx playwright show-report
   ```

5. Tear down:
   ```bash
   cd deployments
   docker compose -f docker-compose.test.yml down --volumes
   ```

### Authentication in E2E Tests

The backend runs with `DEV_MODE=true` in the test stack, which enables a "Dev Login (Admin)" button on the login page. E2E tests that need authentication use this flow:

```ts
// Direct approach (used in audit.spec.ts):
await page.goto('/login');
const devLoginBtn = page.getByRole('button', { name: 'Dev Login (Admin)' });
await devLoginBtn.click();
// ... wait for redirect

// Fixture approach (used in oidc-settings.spec.ts):
import { test, expect } from '../fixtures/auth';
test('page loads', async ({ loggedInPage: page }) => {
  // page is already authenticated
});
```

### Adding a New E2E Test

1. Create `e2e/tests/your-feature.spec.ts`.
2. If the test needs authentication, use the dev login approach shown above.
3. Use Playwright's locator API with role-based selectors (`getByRole`, `getByText`) for resilience.
4. Keep tests independent -- each test should set up its own state.
5. Run with `npx playwright test tests/your-feature.spec.ts` to iterate locally.

## CI Integration

Tests run automatically in `.github/workflows/ci.yml`:

- **unit-test** job: runs `npm run test:coverage`, uploads the coverage report as an artifact.
- **e2e-gated** job: starts the Docker Compose test stack, runs Playwright, uploads the report. This job runs on pushes to `main`, `workflow_dispatch`, and `workflow_call` (from release).

Coverage reports and Playwright HTML reports are available as downloadable artifacts on each workflow run.
